**Purpose:** This matrix defines the goal-state user scenarios that TALA should satisfy when complete. It is an acceptance-test tracker, not proof that every feature is already implemented.  

---

## Scope and Use

This document includes implemented, in-progress, and envisioned scenarios derived from the current Functional Specification, Technical Specification, module capability map, workflow reconciliation matrix, SDD execution map, and current implementation evidence as of 2026-06-19.

The 2026-06-21 specification hardening pass covers benchmark Feature Groups 1-11 through the existing role-oriented AUTH, APP, STU, REG, ACC, FAC, AH, SSA, and OPS scenarios. FS §2.3 and TS §1.4 are the compact goal-state trace layer. A feature-group baseline marked complete means its expected behavior is specified; it does not mean every row is currently executable or passed.

The Pass and Fail columns are intentionally blank until formal execution. The `Comments/Suggestion` column records the current implementation state or dependency note.

For the 2026-06-21 rescue period, rows must be interpreted in two layers:

- **Goal-state row:** a valid envisioned requirement that may remain in this master matrix.
- **Testable-now row:** a row that can be executed against the current committed/demo build through visible UI, direct route, or controlled automated/integration evidence.

Only testable-now rows may receive Pass/Fail marks for the presentation build. Goal-state or execution-gated rows remain blank and must be marked Blocked/In Progress/Deferred in Comments instead of being treated as failed implementation proof.

UI expectations follow the approved framework split:

- Student, applicant, and public-facing goal state: Laravel Livewire, TallStackUI, Tailwind CSS, Alpine.js, and PWA read-only caching where allowed.
- Staff goal state: Filament Admin Nexus resources, pages, tables, forms, infolists, widgets, and typed actions guarded by role/permission policies.

## Local Implementation and Test Evidence Snapshot

This baseline reflects both goal-state requirements and local implementation evidence. Current tests already cover major backend/admin contracts, including `StudentHubAccessTest`, `StudentDashboardServiceTest`, `ApplicantIntakeServiceTest`, `DocumentUploadReviewServiceTest`, `SDD07AAdmissionSetupFilamentResourceTest`, `SDD07AAdmissionReadinessDashboardTest`, `TAL10RbacMatrixTest`, `TAL12A*FilamentResourceTest`, `PaymentConfirmationServiceTest`, `StudentEnrollmentServiceTest`, `PromissoryNoteLifecycleServiceTest`, and scheduling workflow tests.

Current Student Hub evidence is mixed: authenticated Student Hub routes and placeholder Livewire pages exist, and the dashboard aggregate service is tested, but several UI tabs still show placeholder/static content. Current PWA evidence exists through `erag/laravel-pwa`, `config/pwa.php`, `public/manifest.json`, `public/sw.js`, `public/offline.html`, `public/serviceworker.js`, Student Hub layout directives, and icon/splash assets. PWA install/splash/loading scenarios below are therefore goal-state acceptance rows until browser/device smoke tests are executed.

### UI-Reality Rule for Implemented Surfaces

- Every page visible in the effective role navigation must map to at least one case in this master, even when the page is only partially complete.
- A navigation/render case proves that the role can open the page and see the controls allowed by current authorization. It does not prove every business action on that page.
- Business actions exposed by the current Filament, Livewire, or Blade code require their own functional, negative, security, or edge case where applicable.
- Goal-state rows remain valid requirements, but their Comments/Suggestion cell must not describe an unavailable UI as implemented.
- The dated UI traceability audit under `uat-readiness/` is the page-to-case index for the locked submission baseline.

### Exact-Step Execution Standard

Every test row must identify the executing role/account, starting route or page, named control, fixed UAT input, observable result, and the record/count/state that must remain unchanged on failure. A tester must not invent a route, choose an arbitrary record, or interpret phrases such as `if available`, `as appropriate`, `representative record`, or `current policy` while executing a locked case.

- When UI exists, the Action cell names its current label and route from code evidence.
- When UI does not exist, Step 1 is an `Execution gate`; the tester records `Blocked`, leaves Pass/Fail blank, and stops rather than simulating a pass.
- When an action is backend-only or requires a controlled outage/fake, the row states the automated/integration prerequisite and cannot be marked as a manual UI pass.
- Linked cases provide prerequisites only when their exact test ID is named. The tester records generated record IDs in Comments/Suggestion without replacing the fixed input values.
- A response may be 403, 404, or an authorized redirect only where the purpose is denial; the tester records the actual HTTP/final URL and verifies that no protected data rendered before denial.
